# Design Decisions

## 为什么选择 BM25 + 向量混合，而不是纯向量

SQLAlchemy 文档里有很多强术语约束的内容，比如 `joinedload`、`AsyncSession`、`select()`、`sessionmaker`。这类词既有明确语义，也有很强的精确匹配需求。纯向量检索在泛化表达上更强，但遇到专有名词、版本迁移术语、API 名称时，容易把“语义接近但并不对题”的段落排到前面。因此当前实现先用向量召回保留语义覆盖，再用 BM25 和标题 / section 命中把术语相关性拉回来，最后再做轻量 rerank。

这个组合的目标不是追求最复杂的检索链路，而是让“开发文档问答”这种高术语密度场景更稳。面向面试展示时，这也能更清楚地说明：项目不是简单接一个 embedding，而是针对文档类型做了检索策略设计。

## 为什么 chunk 按 section 切分，而不是固定 token 窗口

开发文档天然就有章节结构，尤其 SQLAlchemy 这类官方文档中，标题、说明段落、示例代码、注释通常围绕同一个 section 组织。按 section 切分可以最大程度保留语义完整性，也更容易把 citation 精确映射回具体章节；如果直接按固定 token 窗口切，常见问题是一个代码示例被切断、标题和正文脱节，或者多个不相干段落被拼到同一个 chunk 里。

这次实现里还顺手修了去重提取逻辑，避免 `pre/code/li` 这种嵌套节点重复进入 chunk。这样索引质量和后续生成质量会更稳定，也更适合在技术亮点页里做 chunk 可视化展示。

## 为什么要做 section anchor，而不是只返回整页 URL

RAG 项目里“能引用”不等于“引用可信”。如果回答只附一个整页 URL，用户还得自己翻很久去找具体段落，citation 的说服力会被明显削弱。把 chunk URL 直接做成 `...html#section-id` 之后，用户点击就能跳到对应章节，这会显著提升可核验性，也让引用更像工程产品而不是调试日志。

从展示角度看，section anchor 也是一个很适合讲给面试官听的细节：它说明这个项目不仅在做检索准确率，也在认真处理回答的可追溯性和用户体验。
